# 大纲与段落审阅工作台设计

## 背景

当前 MVP 已经支持 DOCX 上传、段落解析、默认选择待润色段落、润色、导出和下载，但用户看不到解析后的论文结构，也不能在润色前逐段确认。下一阶段要补齐“先分析结构，再由用户审阅确认，最后才开始润色”的核心工作流。

本规格只覆盖大纲与段落审阅工作台，不包含格式保持型 DOCX XML 替换、真实模型效果优化、DOC 转换器和长期生产部署。

## 目标

- 解析完成后展示树形大纲和段落列表。
- 用户能看到每个段落的类型、风险、跳过原因、编号和引用数量。
- 用户能勾选或取消勾选正文与摘要段落。
- 标题、短标签、参考文献、关键词等不可润色内容显示为跳过，不允许勾选。
- 点击“开始润色”前，系统使用数据库中的最新勾选状态。
- 电脑端采用左侧大纲树、右侧段落审阅；手机端退化为上方章节筛选、下方段落卡片。

## 非目标

- 不做段落正文编辑器，第一版只允许勾选是否润色。
- 不在该阶段重做 DOCX 导出格式保持。
- 不承诺第三方检测平台分数。
- 不做多人协作和实时同步。

## 推荐方案

采用“左树右段落”方案。

电脑端布局：

- 顶部保留任务名称、状态、进度和操作按钮。
- 左侧展示按标题聚合的树形大纲。
- 右侧展示当前章节下的段落卡片。
- 右侧顶部展示统计：总段落、已选、跳过、需人工确认。

手机端布局：

- 大纲树变为横向章节筛选或折叠选择区。
- 段落卡片单列排列。
- 底部或任务卡片内提供“保存选择”和“开始润色”按钮。

## 数据模型

沿用现有 `ParagraphRecord`：

- `outlinePath`：所属大纲路径或段落路径。
- `index`：全文顺序。
- `type`：`heading`、`abstract`、`keywords`、`reference`、`body`、`skipped`。
- `selected`：是否参与润色。
- `status`：`selected`、`skipped`、`validated`、`needs_manual_decision` 等。
- `skipReason`：跳过原因。
- `riskLevel`：风险等级。
- `citationCount`：引用标记数量。
- `numberingPrefix`：编号前缀。

第一版不新增 Prisma 字段。大纲树由 `paragraphs` 在服务端或客户端派生：

- `heading` 段落成为章节节点。
- 非 heading 段落归属到最近的 heading 节点。
- 如果文档开头没有 heading，则归入“未分章节”节点。

## API 设计

新增接口：

`PATCH /api/tasks/[taskId]/paragraphs`

请求体：

```json
{
  "selections": [
    { "id": "paragraph-id", "selected": true }
  ]
}
```

行为：

- 只允许更新属于该任务的段落。
- 对 `heading`、`keywords`、`reference`、`skipped` 类型强制保持 `selected=false`。
- 对 `body` 和 `abstract` 类型允许更新。
- 返回更新后的 task，包含按 `index asc` 排序的 paragraphs。

现有接口调整：

- `POST /api/tasks/[taskId]/rewrite` 继续从数据库读取 `selected=true` 的段落。
- 前端点击开始润色前先保存选择；保存成功后再调用 rewrite。

## 组件设计

新增 `TaskReview`：

- 接收完整 task。
- 派生章节树和段落统计。
- 管理本地勾选状态。
- 提供保存选择和开始润色动作。

新增 `OutlineTree`：

- 展示章节节点、段落数量、已选数量。
- 点击章节后筛选右侧段落。
- 手机端可以渲染为横向按钮组。

新增 `ParagraphReviewList`：

- 展示段落卡片。
- 可选段落显示 checkbox。
- 不可选段落显示“跳过”标记和原因。
- 显示类型、风险、引用数量、编号保护提示。

调整 `TaskList`：

- 对 `awaiting_review` 任务展开 `TaskReview`。
- “开始润色”按钮由 `TaskReview` 接管，避免用户没有保存选择就进入润色。

## 用户流程

1. 用户上传 DOCX。
2. 用户点击“解析大纲”。
3. 系统解析段落并进入 `awaiting_review`。
4. 页面展开大纲与段落审阅工作台。
5. 用户按章节查看，勾选或取消勾选段落。
6. 用户点击“保存选择”或“确认并开始润色”。
7. 系统保存选择。
8. 系统开始润色，只处理 `selected=true` 的段落。
9. 进入导出流程。

## 错误处理

- 保存选择失败：保留本地勾选状态，显示错误文案，不进入润色。
- rewrite 失败：任务进入 `failed` 或保持现有失败处理，不吞掉错误。
- 段落为空：显示空态“暂无可审阅段落，请重新解析或上传其他文档”。
- 全部取消选择：允许保存，但“开始润色”按钮禁用，并提示至少选择一个段落。

## 测试计划

单元测试：

- 大纲树派生：无标题、多个标题、标题前摘要、参考文献跳过。
- 选择更新接口：只更新当前任务段落。
- 不可选类型强制保持 `selected=false`。

组件测试：

- `awaiting_review` 任务显示审阅工作台。
- 勾选段落后统计变化。
- 跳过段落不可勾选且显示原因。

端到端 QA：

- 上传测试 DOCX。
- 解析大纲。
- 取消勾选一个正文段落。
- 开始润色。
- 验证被取消的段落没有被润色。
- 验证手机端布局不溢出。

## 验收标准

- 用户能清楚看到论文大纲树。
- 用户能在润色前审阅并修改待润色段落集合。
- 跳过内容可见、原因明确、不可误选。
- 润色接口严格尊重用户最终选择。
- 桌面和手机端都能完成审阅与开始润色。
- 所有新增测试通过，现有测试不回退。
